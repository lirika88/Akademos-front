## Примерный CI/CD Pipeline проекта (Рабочий процесс)

Так как проект выполняется командой из двух человек (я — фронтенд, напарник — бэкенд), процесс разработки клиентской части организован следующим образом:

*   **Этап проектирования интерфейсов:** На основе макетов (или требований) создаю HTML-страницы и CSS-стили. Для имитации работы с сервером на этом этапе используются мок-данные (JSON-объекты, вшитые в JavaScript-код), чтобы увидеть, как будут выглядеть списки учеников, тесты и т.д.
*   **Согласование API:** Мы с напарником фиксируем форматы данных (какие поля приходят от сервера, какие отправлять обратно). Это позволяет мне писать JS-логику, не дожидаясь готовности сервера.
*   **Интеграция:** После того как бэкенд-часть готова, я заменяю мок-данные на реальные вызовы Fetch API к серверу напарника. На этом этапе происходит отладка взаимодействия.
*   **Система контроля версий:** Для хранения кода используется Git. В репозитории `Akademos` ведутся две основные ветки: `main` (стабильная версия) и `develop` (версия для разработки). Каждая новая функциональность (например, страница "Конструктор тестов") разрабатывается в отдельной ветке `feature/...`. После завершения работы создается Pull Request для слияния в `develop`.
*   **Деплой (Публикация):** Так как проект представляет собой набор статических файлов (HTML, CSS, JS), для его публикации используется сервис Netlify. При пуше в ветку `main` происходит автоматическое обновление работающей версии сайта.

### Детальный рабочий процесс

1.  **Получение задачи**
    *   Разработчик получает задачу (например, разработка страницы учеников)
2.  **Создание ветки в Git**
    *   Выполняется команда `git checkout -b feature/students-page`
3.  **Разработка компонентов**
    *   Создается HTML-разметка страницы
    *   Пишутся CSS-стили
    *   Разрабатывается JavaScript-логика с использованием мок-данных
    *   Проводится проверка в браузере
4.  **Фиксация изменений**
    *   Выполняется `git add .`
    *   Создается коммит `git commit -m "Добавлена страница учеников"`
    *   Изменения отправляются в репозиторий `git push origin feature/students-page`
5.  **Создание Pull Request**
    *   На GitHub создается запрос на слияние из ветки `feature/students-page` в `develop`
    *   Добавляется описание выполненных изменений
6.  **Автоматическая проверка**
    *   Запускаются линтеры для проверки кода
    *   Проверяется корректность сборки проекта
7.  **Слияние веток**
    *   После успешной проверки нажимается кнопка "Merge"
    *   Код попадает в ветку `develop`
8.  **Деплой на тестовый сервер (Staging)**
    *   Netlify автоматически разворачивает тестовую версию
    *   Разработчик проверяет работоспособность по тестовой ссылке
9.  **Релиз готовой версии**
    *   Создается Pull Request из `develop` в `main`
    *   После проверки выполняется слияние
10. **Деплой на боевой сервер (Production)**
    *   Netlify автоматически разворачивает финальную версию
    *   Сайт становится доступен пользователям по основной ссылке
